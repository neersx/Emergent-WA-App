WhatsApp Business Management Platform

A multi-tenant SaaS platform for managing WhatsApp Business Accounts — templates, inbound conversations, outbound messaging, and per-message cost tracking — built on the FARM stack.

Phase 1–2.5 Complete FARM Stack Live Mode: META_MOCK_MODE=false Meta Graph API v21.0 July 2026

1 Executive Summary

REST endpoints

The platform enables Technology Providers to onboard client businesses onto WhatsApp via Meta's Embedded Signup flow, manage their message templates, send outbound template messages, receive and reply to inbound messages in a real-time conversation inbox, and track per-message costs with daily rollup billing reports — all with strict multi-tenant isolation.

Functional Scope
Multi-tenant auth (JWT + MFA) · Embedded Signup (WABA onboarding) · Template CRUD + Meta Graph submit · Real-time inbox (WebSocket) · Outbound template sending · Cost capture & billing dashboard · Platform super-admin (cross-tenant) · CSV/XLSX export
Correctness Guarantees
24h service-window enforced server-side · Webhook ACK <1s (async worker) · Fernet-encrypted business tokens · HMAC-SHA256 webhook verification · Tenant RLS on every query · Elevated cross-tenant access audited

Row-Level Security (RLS) Implementation Every MongoDB query in every route handler includes {"tenant_id": p.tenant_id} from the Principal object extracted from the JWT. The only exception is platform.py which does explicitly audited elevated queries — not by disabling the filter globally, but by calling _audited() before any cross-tenant read.

5 Security Architecture

Authentication

1
Password loginbcrypt (rounds=10) — passwords never stored in plaintext
2
Access tokenJWT HS256 · 15-minute TTL · carries sub, role, tenant_id, mfa_complete
3
Refresh tokenRandom 48-byte opaque string · SHA-256 hash stored in MongoDB · 14-day TTL · httpOnly Secure cookie · rotated on every refresh
4
MFA (TOTP)pyotp · TOTP secret encrypted at rest with Fernet · required for PlatformSuperAdmin
5
WebSocket authhttpOnly access_token cookie automatically sent by browser in WS handshake headers — server reads via FastAPI Cookie() dep

Credential Storage

1
Meta business tokensFernet (AES-128-CBC + HMAC-SHA256) encrypted before MongoDB insert. Decrypted in-memory only at send time. NEVER returned to frontend.
2
Webhook signatureHMAC-SHA256 of raw request body (before JSON parse) validated against META_APP_SECRET. Rejects any request with invalid or missing signature.
3
Media URLsHMAC-SHA256 signed · 1-hour TTL · path + expiry in signature · never served publicly
4
Password reset48-byte random token · SHA-256 hash stored · 30-min TTL · single-use (marked used on consume)

8 Webhook Processing Flow

Critical correctness rule: The webhook endpoint MUST ACK in under 1 second. All heavy processing (projection, cost capture, media download, WebSocket broadcast) happens asynchronously in the background worker — never in the request handler.
Meta Webhook → ACK → Worker → Broadcast
sequenceDiagram participant Meta as Meta Cloud API participant WH as POST /api/webhooks/meta participant DB as MongoDB participant Q as mq_jobs queue participant W as Background Worker participant WS as WebSocket Manager Meta->>WH: POST (HMAC-SHA256 signed) WH->>WH: verify_signature(raw_body, X-Hub-Signature-256) alt Invalid signature WH-->>Meta: 401 Unauthorized else Valid WH->>DB: insert webhook_events_raw {payload, signature_ok, status:"received"} WH->>Q: enqueue("process_webhook", {event_id}) WH-->>Meta: 200 OK {received:true} ← within 1 second end Note over W: Worker polls every 500ms W->>Q: claim_one() — atomic find_one_and_update Q-->>W: job doc {event_id} W->>DB: find webhook_events_raw by event_id DB-->>W: raw payload loop For each entry.changes[] alt field == "messages" W->>W: parse statuses[] + messages[] W->>DB: update_one messages {status, timestamps, pricing} W->>DB: upsert conversations {last_inbound_at, unread_count} W->>DB: insert_one messages (inbound) W->>DB: upsert usage_daily_rollup (cost capture) W->>WS: broadcast(tenant_id, {type:"new_message"|"status_update"}) else field == "message_template_status_update" W->>DB: update_many templates {status, rejection_reason} W->>WS: broadcast(tenant_id, {type:"template_update"}) end end W->>DB: mark webhook_events_raw {status:"processed"} W->>Q: complete(job_id)
