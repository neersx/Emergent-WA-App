1 Why MongoDB?

FeatureWhy it matters in this application
Flexible schemaInbound messages can be text, image, video, location, reaction — each needs different fields. MongoDB stores each as a document without unused null columns.
Native JSON docsMeta webhook payloads arrive as JSON and are stored as-is, preserving nested structure perfectly.
Atomic find_one_and_updateThe job queue claims exactly one job without race conditions — no SELECT FOR UPDATE or distributed locking needed.
TTL indexesRefresh tokens and idempotency keys expire automatically — zero cleanup code required.
Aggregation pipelineAnalytics queries (timeseries, cost rollup, template performance) use $group, $match, $addFields — no separate analytics DB.
Upsert + $setOnInsertWABA onboarding, conversation tracking, and cost rollup are fully idempotent using upsert — safe to call multiple times.
Motor async driverFastAPI is fully async; Motor provides native async/await MongoDB I/O with zero thread-pool overhead.

4 MongoDB Core Concepts

Hierarchy
MongoDB Server (mongod)
└──
Database: "whatsapp_saas"
└──
Collection: "messages"
← like a SQL table
└──
Document: { "_id": "...", "body": "..." }
← like a SQL row
└──
Field: "body": "Hello world"
← like a SQL column

Key Differences vs SQL

ConceptSQLMongoDB
TableCREATE TABLE users (...)Collection db.users
RowFixed columns, nullableFlexible JSON document
Primary keyid SERIAL PRIMARY KEY_id field
Foreign keyREFERENCES tenants(id)Field by convention (e.g. tenant_id)
SchemaEnforced by DDLOptional (by application)
Auto-expire rowspg_cron / scheduled jobsTTL Index — zero code needed
AggregationGROUP BY, SUM()$group, $sum in pipeline
This app's _id convention: Uses UUID strings instead of MongoDB's default ObjectId. This makes IDs URL-safe and portable. The _id and id fields always hold the same UUID value — _id is MongoDB's primary key, id is the API-facing name.

9 How the App Connects — Motor Driver

backend/.env
MONGO_URL=mongodb://localhost:27017
DB_NAME=whatsapp_saas
python — app/db.py (connection setup)
from motor.motor_asyncio import AsyncIOMotorClient, AsyncIOMotorDatabase

_client: AsyncIOMotorClient | None = None
_db: AsyncIOMotorDatabase | None = None

def get_db() -> AsyncIOMotorDatabase:
    global _client, _db
    if _db is None:
        _client = AsyncIOMotorClient(os.environ["MONGO_URL"])
        _db = _client[os.environ.get("DB_NAME", "whatsapp_saas")]
    return _db
python — using the db in a route
from .db import get_db

@router.get("/conversations")
async def list_conversations(p: Principal = Depends(require_tenant)):
    db = get_db()                          # get the db handle
    docs = await db.conversations           # access a collection
        .find({"tenant_id": p.tenant_id})   # ALWAYS filter by tenant (RLS)
        .sort("last_inbound_at", -1)       # most recent first
        .limit(50)
        .to_list(50)                        # execute → Python list
    return docs

10 Key Query Patterns

Pattern 1 — Tenant-Scoped Query (RLS)

python
# CORRECT — always scope to tenant_id from JWT
docs = await db.messages.find({"tenant_id": p.tenant_id}).to_list(50)

# WRONG — never query without tenant scope
docs = await db.messages.find({}).to_list(50)  # returns ALL tenants' data!

Pattern 2 — Idempotent Upsert

python
await db.conversations.update_one(
    {"id": conv_id, "tenant_id": tenant_id},   # filter (identity)
    {
        "$set": {"last_inbound_at": now},      # runs EVERY time
        "$inc": {"unread_count": 1},           # runs EVERY time
        "$setOnInsert": {                      # runs ONLY on first insert
            "_id": conv_id,
            "status": "open",
            "created_at": now,
        },
    },
    upsert=True,
)

Pattern 3 — Atomic Job Claim

python
job = await db.mq_jobs.find_one_and_update(
    {
        "$or": [{"status": "queued"},
               {"status": "processing", "claimed_at": {"$lt": stale_cutoff}}],
        "attempts": {"$lt": 5},
    },
    {"$set": {"status": "processing", "claimed_at": now},
     "$inc": {"attempts": 1}},
    sort=[("created_at", 1)],  # FIFO order
    return_document=True,
)
# Only ONE worker ever gets this job — guaranteed by MongoDB atomicity

11 Aggregation Pipelines

mongosh — template performance stats
db.messages.aggregate([
  {$match: {tenant_id:"fc26e509-...", direction:"outbound", template_name:{$ne:null}}},
  {$group: {
    _id: "$template_name",
    sent: {$sum: 1},
    delivered: {$sum: {$cond:[{$in:["$status",["delivered","read"]]},1,0]}},
    read: {$sum: {$cond:[{$eq:["$status","read"]},1,0]}}
  }},
  {$sort: {sent: -1}}
])
mongosh — daily timeseries
db.messages.aggregate([
  {$match: {tenant_id:"fc26e509-...", created_at:{$gte:"2026-06-01"}}},
  {$addFields: {day: {$substr:["$created_at",0,10]}}},
  {$group: {_id:{day:"$day",dir:"$direction"}, count:{$sum:1}}},
  {$sort: {"_id.day":1}}
])

13 TTL (Auto-Expiry) Indexes

How TTL works: MongoDB's background TTL monitor runs every 60 seconds. It deletes any document where field_value + expireAfterSeconds ≤ current time. Zero cron jobs or application cleanup code needed.
CollectionTTL fieldExpires afterPurpose
refresh_tokensexpires_atWhen expires_at passes (14 days)Auto-logout idle sessions
password_resetsexpires_atWhen expires_at passes (30 min)Expire reset tokens
idempotency_keyscreated_at24 hours after creationKeep dedup window to 24h only
mongosh — verify TTL index
// Confirm TTL is configured
db.refresh_tokens.getIndexes()
// Look for: "expireAfterSeconds": 0

// Manual cleanup (if TTL hasn't run yet)
db.refresh_tokens.deleteMany({expires_at: {$lt: new Date()}})

14 Useful Maintenance Queries

mongosh — health & stats
// Connection health
db.runCommand({connectionStatus: 1})

// Count docs in every collection
db.getCollectionNames().forEach(c => {
  print(c + ": " + db[c].countDocuments())
})

// Storage per collection
db.getCollectionNames()
  .map(c => ({name:c, size: db[c].stats().storageSize}))
  .sort((a,b) => b.size - a.size)
  .forEach(x => print(x.name + ": " + (x.size/1024).toFixed(1) + " KB"))

// Clear completed jobs (keep last 7 days)
const week = new Date(Date.now()-7*86400000).toISOString()
db.mq_jobs.deleteMany({status:"done", completed_at:{$lt:week}})

// Full reset (dev only!) ⚠️
db.dropDatabase()
// Restart the backend — it auto-re-seeds users + demo data

15 Backup & Restore

bash — backup
# Full backup to a directory
mongodump \
  --uri="mongodb://localhost:27017/whatsapp_saas" \
  --out="./backup/$(date +%Y-%m-%d)"

# Compressed single-file backup
mongodump \
  --uri="mongodb://localhost:27017/whatsapp_saas" \
  --gzip \
  --archive="./backup/whatsapp_saas_$(date +%Y-%m-%d).gz"
bash — restore
# Restore from directory
mongorestore \
  --uri="mongodb://localhost:27017" \
  --db="whatsapp_saas" \
  "./backup/2026-07-05/whatsapp_saas/"

# Restore from compressed archive
mongorestore \
  --uri="mongodb://localhost:27017" \
  --gzip \
  --archive="./backup/whatsapp_saas_2026-07-05.gz"

17 Common Errors & Fixes

ServerSelectionTimeoutError: localhost:27017 MongoDB is not running. Fix: brew services start mongodb-community@7.0
WriteError: Updating the path 'unread_count' would create a conflict You used both $inc and $setOnInsert on the same field. Remove the field from $setOnInsert — $inc will set it to 1 on insert automatically.
DuplicateKeyError: E11000 duplicate key Violates a unique index. Use upsert instead of insertOne, or check which index is violated with db.collection.getIndexes()
ValidationError: value is not a valid Fernet key The TOKEN_ENCRYPTION_KEY in .env is wrong. Regenerate: python3 -c "import base64,secrets; print(base64.urlsafe_b64encode(secrets.token_bytes(32)).decode())"
